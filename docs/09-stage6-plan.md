# Stage 6 Implementation Plan — Vocabulary learning system

## Objectives

1. A shared vocabulary "dictionary cache" (definition, example, synonyms, antonyms, CEFR level) generated by the LLM on first lookup and reused after that.
2. A personal vocabulary notebook per student.
3. Embeddings-backed similarity lookups (synonym/related-word discovery).
4. Automatic vocabulary recommendations surfaced after a conversation, based on words the student encountered that were above their level.

## Embeddings: fastembed (ONNX) instead of raw sentence-transformers+PyTorch

[docs/02-technology-selection.md](02-technology-selection.md) specifies "sentence-transformers (local ONNX/PyTorch model)". The PyTorch-backed `sentence-transformers` package pulls in a full PyTorch install (600MB–2GB depending on build) just to run a tiny embedding model at inference time — a poor fit for "ease of deployment" on modest school hardware. `fastembed` runs the same class of model (all-MiniLM-L6-v2, ONNX export) via `onnxruntime` directly, with no PyTorch dependency, at a fraction of the install size. This is the ONNX path the tech-selection doc already called out, via a lighter runtime than the full `sentence-transformers` package — not a different architecture, just a smaller way to get the same result.

## Embeddings storage: brute-force cosine similarity instead of sqlite-vec (for now)

The tech-selection doc also specifies `sqlite-vec` for indexed vector search inside SQLite. For Stage 6, embeddings are instead stored as raw blobs (Float32Array) directly on the `vocabulary` table row, and similarity search is done with in-process cosine similarity over however many words are in the dictionary cache.

This is a deliberate, scale-aware simplification: a single school's shared vocabulary dictionary is expected to stay in the hundreds-to-low-thousands of rows (every distinct word the AI has ever explained across all students), which is trivial to brute-force compare against in a few milliseconds — an indexed ANN structure has no measurable benefit at this size and adds a native-extension-loading dependency (with the attendant "does the prebuilt binary work on this exact platform" risk that already bit the Android SDK in Stage 2) for no real gain yet. `sqlite-vec` remains the documented upgrade path if a district-scale multi-school deployment (Stage 10+ territory) ever needs it — the `VocabularySimilaritySearch` interface (see below) is written so swapping the implementation later doesn't change any caller.

## Database schema

- `vocabulary` (shared cache, not per-student): id, word (unique), definition, example, synonyms (JSON array as text), antonyms (JSON array as text), cefr_level, embedding (BLOB, Float32Array), created_at.
- `vocabulary_notebook` (per-student): id, student_id, vocabulary_id, source (`manual` | `recommended`), created_at. Unique on (student_id, vocabulary_id) — adding the same word twice is a no-op, not a duplicate row.

## AI Service additions

- `POST /v1/vocabulary/explain` — body `{ word, difficultyLevel }`. Non-streaming (short, structured output, same reasoning as grammar/explain). Returns `{ definition, example, synonyms: string[], antonyms: string[], cefrLevel }`. Uses the same lenient marked-section parsing pattern as `/v1/grammar/explain` rather than requiring strict JSON from a small local model.
- `POST /v1/embed` — body `{ text }`. Returns `{ embedding: number[] }` (384-dim for all-MiniLM-L6-v2) via `fastembed`.

## Backend routes

- `POST /vocabulary/lookup` — body `{ word }`. Returns the cached `vocabulary` row if present; otherwise calls the AI service's explain + embed endpoints, caches the result, and returns it. Available to any authenticated role (a teacher or admin exploring a word doesn't need to "own" it the way a conversation belongs to a student).
- `POST /vocabulary/notebook` — body `{ word }`. Looks up/generates the vocabulary entry (reusing the lookup logic above) and adds it to the caller's notebook.
- `GET /vocabulary/notebook` — the caller's notebook, joined with full vocabulary detail.
- `DELETE /vocabulary/notebook/:id` — removes a notebook entry (ownership-checked).
- `GET /vocabulary/recommendations?conversationId=` — ownership-checked against the conversation. Scans the assistant's messages in that conversation for candidate "difficult" words (a v1 heuristic: word length + a small common-word stoplist, excluding words already in the student's notebook — see below), looks up/generates vocabulary entries for the top few candidates, and returns them as recommendations. Recommendations are *shown*, not auto-added — the student chooses what to save, matching "the system recommends, the student curates their own notebook."
- `GET /vocabulary/similar?word=` — cosine-similarity search over the `vocabulary` table's stored embeddings, returning the closest other cached words. This is the concrete embeddings-pipeline deliverable ("synonym/antonym/recommendation lookups" from the original requirements) — note it searches words the system already knows about (from lookups/notebooks across the school), not the entire English language, which is an honest scope limit of a locally-cached dictionary rather than a general thesaurus.

## Difficult-word heuristic (v1)

Candidate words are pulled from the assistant's replies in a conversation (not the student's own messages — the point is exposing the student to words *above* their level, which by definition they didn't produce themselves). A word qualifies as a candidate if: it's 7+ characters, alphabetic, not in a ~100-word common-English stoplist, and not already in the student's notebook. This is intentionally crude (documented as v1, same posture as the Stage 4 difficulty heuristic) — a better version would use actual frequency-list data or the word's own generated `cefrLevel`, which becomes available for a word only after it's already been looked up once; that refinement is a natural Stage 8 (analytics) follow-up once there's usage data to justify it.

## Desktop UI

- A "Vocabulary Notebook" section in the sidebar: add-word input (manual lookup + add), list of saved words, remove button, click-to-expand full detail (definition/example/synonyms/antonyms/CEFR).
- After an AI reply in a conversation, a "Words to learn" strip appears if `/vocabulary/recommendations` returns any candidates, each with an "Add to notebook" button.

## Testing procedure

1. Backend unit tests fake the AI service's explain/embed calls (same pattern as Stages 4–5) so vocabulary-cache-and-lookup, notebook CRUD, the difficult-word heuristic, and cosine similarity ranking are all tested without a live model.
2. Manual end-to-end: look up a real word through the live AI service (confirm definition/example/synonyms/antonyms/CEFR and a real embedding vector come back), add it to a notebook, have a conversation, confirm recommended words appear, add one, confirm `/vocabulary/similar` returns sensible neighbors once a few words are cached.
